# Gauntleet

LeetCode-style algorithm practice site where problems are generated by LLMs. The interesting bit isn't generation — it's **verification**: a generator model writes the problem and a reference solution, a validator model independently solves it, and we cross-check both on random inputs from a generated input function. Disagreement means the problem is thrown out before it ever reaches a user.

Local-first MVP, intended as a portfolio piece. Stack: Next.js 15 + TypeScript, SQLite via Drizzle, Docker-isolated Python sandbox per submission, pluggable LLM providers (Anthropic + OpenAI-compatible covering OpenAI, LM Studio, Ollama).

## Repo layout

```
apps/web/              # Next.js app (UI lands in PR #6)
packages/
  core/                # Generation orchestration: prompt, zod schema, python-harness, generate()
  llm/                 # LLMProvider abstraction (anthropic + openai-compatible)
  sandbox/             # Docker-based Python execution
  db/                  # Drizzle + SQLite schema, migrations, migrate()
docker/python-runner/  # Sandbox image (Dockerfile + runner.py harness)
data/                  # SQLite file (gitignored)
.env.local             # Local config (gitignored, repo root — NOT under apps/web)
```

## Conventions

- **Use `pnpm`, not `npm` or `yarn`.** The workspace relies on pnpm-only filter syntax (`pnpm --filter @gauntleet/core …`) and pnpm's lockfile. Mixed package managers will break installs.
- **Feature branch + PR workflow.** Never commit straight to `main`. Each logical step gets its own branch (`feat/foo` or `chore/foo`) and merges via a PR so reverts stay clean.
- **Tests are required with feature PRs.** Every feature PR ships with Vitest unit tests for new pure-logic code. Sandbox/LLM/DB integration is exercised by the existing smoke scripts — don't backfill mocks for those, it just creates test drift.
- **Don't add doc files unless asked.** This file (CLAUDE.md) is the only project-wide markdown that's expected; PRs and commit messages carry the rest.
- **Strict TypeScript.** Root `tsconfig.base.json` has `strict: true` and `noUncheckedIndexedAccess: true`. Don't relax these.

## Architecture notes worth remembering

- **LM Studio rejects `response_format: { type: "json_object" }`** — it only accepts `json_schema` or `text`. We use prompt-based JSON for both providers (a "reply with a single valid JSON object" reminder appended to the system message). Don't reintroduce `response_format` without provider-specific branching. The `extractJSON()` helper in `packages/core/src/generate.ts` is tolerant of fenced code blocks and surrounding prose.
- **Cross-validation requires independent model families.** `checkIndependence(generator, validator)` warns when they share a `familyId` (anthropic = `"anthropic"`, openai-compatible = the base URL). Same-family pairs effectively self-validate and won't catch generator mistakes. The smoke script and the eventual UI both surface this warning — don't silence it.
- **The sandbox is intentionally dumb.** It just executes Python with isolation flags and pipes stdin/stdout. Anything problem-shaped (function dispatch, JSON marshalling, arg unpacking) lives in `packages/core/src/python-harness.ts`. Reuse that harness for any new code-execution use case (validator solutions, user submissions, etc.).
- **Sandbox isolation is non-negotiable.** Every `runPython()` call uses: `--network=none`, `--read-only`, tmpfs `/tmp` + `/sandbox`, `--memory` + `--memory-swap` (no swap), `--cpus`, `--pids-limit`, `--cap-drop=ALL`, `--security-opt no-new-privileges`, `--user 65534:65534`, plus a host-side `docker kill` timer. Don't loosen these without thinking through the threat model.
- **Output is capped at 1 MiB per stream.** User programs writing gigabytes won't OOM the host.
- **`.env.local` lives at the repo root, not in `apps/web/`.** The smoke scripts and the `gen` CLI explicitly load `<repo-root>/.env.local`. Next.js will need similar wiring when the web UI lands.
- **Drizzle migrations are committed.** They live in `packages/db/drizzle/`. The CLI calls `migrate(db)` on startup; future Next.js startup should do the same.
- **OOM detection is heuristic.** Container exit code 137 + `!timedOut` ⇒ probably OOM. `docker inspect`'s `OOMKilled` field would be authoritative but requires keeping the container around after exit; not worth the complexity for the MVP.
- **Sandbox cold-start is ~300ms per invocation.** Acceptable for the MVP; if it ever matters, switch to a long-lived container with `docker exec` (filed for the post-MVP issue list).

## Provider configuration

Two roles — **GENERATOR** (writes problems) and **VALIDATOR** (independently solves for cross-check). Each is configured via env vars:

```
{ROLE}_PROVIDER   # preset: anthropic | openai | lmstudio | ollama | custom
{ROLE}_MODEL      # required
{ROLE}_BASE_URL   # optional override; preset has a default
{ROLE}_API_KEY    # optional override; preset has a default
```

Preset defaults:

| preset      | base URL                      | api key                        |
| ----------- | ----------------------------- | ------------------------------ |
| `anthropic` | (uses Anthropic SDK directly) | `ANTHROPIC_API_KEY` from env   |
| `openai`    | `https://api.openai.com/v1`   | `OPENAI_API_KEY` from env      |
| `lmstudio`  | `http://localhost:1234/v1`    | `lm-studio` (any string works) |
| `ollama`    | `http://localhost:11434/v1`   | `ollama` (any string works)    |
| `custom`    | must supply `BASE_URL`        | must supply `API_KEY`          |

See `.env.example` for the full template.

## CI

Both must be green before merging.

## Pitfalls Claude Code agents have hit before

- Spreading `{ ...DEFAULTS, ...opts }` when `opts` properties are `undefined` overrides defaults with `undefined`. Use `opts.x ?? DEFAULTS.x` instead.
- `z.unknown()` makes object properties optional in inferred types. If the DB schema requires the field, cast at the insert site.
- `next lint` is deprecated as of Next.js 15.5; use `eslint .` directly.
- pnpm doesn't hoist `eslint-plugin-*` for `eslint-config-next` by default. The `.npmrc` has `public-hoist-pattern[]=*eslint*` to fix this — don't remove it.
- `next-env.d.ts` is auto-generated and uses triple-slash references that ESLint flags. It's in `apps/web/eslint.config.mjs`'s ignore list — keep it there.
